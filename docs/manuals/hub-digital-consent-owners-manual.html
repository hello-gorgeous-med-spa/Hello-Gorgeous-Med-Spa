Digital Consent & Hub Intake

Owner’s protocol manual — Hello Gorgeous Med Spa (Oswego, IL)

Purpose. Run digital consent the same way every time: correct URLs, correct files in the site, database rows in Supabase, and a clear front-desk flow. This document is the checklist; the website code is the system of record for paths and slugs.

1. Principles (do not skip)

  • Verbatim legal / clinical text lives in static HTML under public/docs/… on the website. The app does not rewrite that language in React — it shows the file in an iframe and records signature + attestation.
  • When you change a consent PDF or vendor wording, you replace the HTML file, redeploy the site, and (if the title/slug in the database should change) update Supabase — not the other way around.
  • Matching to the chart uses the mobile number the client enters. The server matches to existing clients (including Fresha-imported rows) or creates a minimal new client. Staff should use the number on file when possible.
Regulatory & legal. This manual describes your technical and operational process only. You remain responsible for appropriate consent, scope of practice, record retention, and any state or board requirements. Have your attorney or medical director review the actual consent documents — not this SOP.

2. What you are operating

Component What it is
Hub intake page Public page on the hub subdomain. Patients open it on their phone; no Hub password is required for /intake.
Form templates (Supabase) Table hg_form_templates — one row per form slug. Iframe consents use an empty schema_json (body = static HTML file).
Submissions (Supabase) Table hg_form_submissions — stores signature name, phone, responses_json (includes consent URL and attestation), timestamp, and optional client_id link.
Team review After Hub login: /hub/intake-submissions lists recent submissions.

5. Database: one-time / when adding devices

Run the combined SQL in Supabase (SQL Editor) from the repository:

supabase/manual/hg_form_templates_bootstrap.sql

That script creates the tables and policies (if missing) and seeds:

  • general-intake (optional structured fields)
  • luxora-consent
  • solaria-co2-consent
  • morpheus8-consent

Re-running is idempotent for the consent rows (ON CONFLICT updates titles). If tables already exist, you can use the smaller file supabase/manual/hg_form_templates_solaria_morpheus8.sql for Solaria + Morpheus8 only.

6. Environment & operations (IT checklist)

  • Production has valid Supabase URL and service role key for server routes (submissions and template lookup).
  • Vercel (or host) deploy includes public/docs/luxora/, public/docs/solaria/, public/docs/morpheus8/ with the real HTML files — not empty folders.
  • Hub subdomain DNS points to the same app as the public site (middleware depends on hub. host for intake).
  • Hub session / password (if enabled) is documented separately — intake stays public; Command Center remains protected.

7. Front-desk and clinical protocol

Before the patient signs

  1. Confirm the correct procedure / device for today’s visit.
  2. Open the matching intake URL (section 3) on a staff tablet or have the patient scan a QR you generated from that URL.
  3. Ask the patient to use the mobile number on file (matches Fresha / chart) so the submission links to the right client.

On the form

  1. Patient scrolls the full consent in the iframe (verbatim document).
  2. Patient checks the attestation that they read the full document and had opportunity to ask questions.
  3. Patient completes name and phone, then submits.
  4. Patient receives a reference code on screen — you may log it in your EMR / paper log if that is your policy.

After submit

  1. Staff with Hub access opens /hub/intake-submissions to verify the row for today’s case.
  2. If the wrong phone was used, treat as a data-quality issue: locate the correct chart and follow your policy for amending or re-signing. Do not rely on the app alone to merge duplicate client records without a defined process.

8. What is stored in responses_json (for audit)

For iframe consents, expect at minimum (names may vary slightly by code version):

  • consent_iframe_url — path to the static HTML that was shown.
  • read_full_consent_attested — true when they checked the box.
  • Luxora may also include legacy fields luxora_consent_iframe_url and read_full_luxora_consent_attested for compatibility.

9. Updating a consent (vendor revision or legal edit)

  1. Obtain the final approved text from your medical director / attorney / vendor.
  2. Replace the HTML file in public/docs/… and deploy. Test the iframe on hub intake before clinic hours.
  3. If the form title in Supabase should change, run an UPDATE on hg_form_templates for that slug or re-run the seed SQL that sets the title.
  4. Do not paraphrase consent text inside React/TSX — that diverges from this protocol.

10. Troubleshooting (quick)

SymptomWhat to check
“Form not found” on load Slug in URL matches a row in hg_form_templates with is_active = true. Typos in ?form= are rejected; default is Luxora.
Blank iframe / 404 in frame File exists in public/docs/… at the path in section 4; filename matches the code map. Redeploy after adding files.
Submit works but no client link Phone field empty or invalid; or client create failed. Check hg_form_submissions.client_id in Supabase.
Apex /intake behaves oddly Use hub.hellogorgeousmedspa.com/intake as the canonical link.
